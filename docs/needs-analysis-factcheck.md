## ⚠️ Реализовано иначе, чем заявлено

- **«Фирменный стиль: тёмно-синяя панель»** — реализован графит + сигнальный оранжевый.
  Замечание к самому документу: фирменный цвет Omnicomm — оранжевый, «тёмно-синий» в ТЗ
  сомнителен; текущий стиль ближе к бренду, чем требование.
- **«Импорт через Excel» для миграции данных** — миграция сделана надёжнее: скрипт
  SQLite→PG (`db:migrate-legacy`, идемпотентный, проверен на копии прода 84 строки 1:1).
- **«Автоматическое создание заявок из обращений по всем каналам»** — создание заявок
  из тикетов есть (авто-заявки по объектам), но сами обращения из каналов сейчас фиксируются
  вручную; авто-приём — см. «не реализовано».

## 🔲 Заявлено, но НЕ реализовано → ЗАКРЫТО 07.07.2026 (пп. 1–3, 6)

Пункты 1, 2, 3 и 6 реализованы и проверены сразу после фактчека:
`/reports/installers` (карточки специалистов: активные/закрытые/просроченные,
среднее прибытие/работа из visit_steps, повторные выезды, качество %),
`/service/sla` (нормативы по 16 типам, авторасчёт due_at при создании заявки),
`/api/jobs/overdue` (авто-простановка «Просрочена» + адресные уведомления
менеджеру заявки и head/boss через очередь; идемпотентен),
импорт SIM из Excel на странице SIM (колонки ICCID/Номер/Тариф, дубли
пропускаются, движения source_type='import'). Осталось из списка: пп. 4–5.

### Исходный список (для истории)

1. **Карточка специалиста с метриками** (среднее время прибытия/выполнения, повторные
   выезды, «качество») — есть только план/факт/суммы в `/reports/workload`. Данные для
   метрик уже копятся (visit_steps с таймстампами) — вопрос одного отчёта.
2. **Нормативы реакции/исполнения по типам заявок + автопростановка просрочки** — `due_at`
   задаётся вручную, статус `overdue` в схеме есть, но джоба автоотметки нет; дашборд
   считает просрочку по `due_at` на лету. Нужен справочник нормативов + cron-джоб.
3. **Адресные уведомления по критичным событиям** — частично: push технику при назначении
   и предупреждения о блокировке клиентам есть; уведомлений руководству/менеджеру о
   просрочках нет.
4. **Интеграция IP-телефонии и мессенджеров с автопривязкой** — журнал звонков/сообщений
   ручной; вебхуки/WhatsApp Business API/автопривязка — бэклог (у нас есть Baileys-мост
   как задел). «Запись разговоров» (Could) — нет.
5. **Power BI** — фид не реализован (бэклог); Excel-выгрузки покрывают ручной сценарий.
6. **Импорт SIM из Excel** — в документе не назван прямо, но заявлен в эталоне Аскан;
   у нас только ручное создание и экспорт.

Пункты 1–3 — быстрые (данные уже в БД), 4–5 — интеграционные, требуют внешних учёток.
