# Архитектура

## Компоненты

```
┌─────────────┐      ┌──────────────┐      ┌──────────────┐
│   Web       │─────▶│   API (REST) │─────▶│  PostgreSQL  │
│  Next.js 16 │      │  CRM + sync  │      │   клиенты,   │
│  дашборд    │◀─────│   worker     │◀─────│  договоры,   │
└─────────────┘      └──────┬───────┘      │  привязки ТС │
                            │              └──────────────┘
                            ▼
                   ┌──────────────────┐
                   │  Omnicomm HTTP   │
                   │  API (дерево ТС, │
                   │   телеметрия)    │
                   └──────────────────┘
```

## Поток данных

1. **Синк-воркер** (раз в `OMNICOMM_SYNC_INTERVAL_HOURS`, по умолчанию 3ч) ходит в Omnicomm API: логин → дерево ТС (флэттим) → агрегаты телеметрии за период.
2. Снапшот кэшируется в Postgres, чтобы UI не дёргал Omnicomm напрямую (копия хрупкая под нагрузкой).
3. **CRM-логика** связывает ТС с клиентом/договором, ведёт воронку сделок.
4. **Web** показывает карточку клиента с актуальным состоянием его парка.

## Интеграция с Omnicomm — правила

- Отчёты — `POST`, не GET.
- Топливо — децилитры, делить на 10.
- Дерево ТС — вложенное, флэттить.
- Health-проба перед каждым забором; не запускать параллельные тяжёлые запросы к одной копии.
