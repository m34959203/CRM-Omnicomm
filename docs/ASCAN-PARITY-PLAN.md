# План разработки: аналог функционала «Аскан: МТ» на базе CRM-Omnicomm

Дата: 05.07.2026. Входные данные: транскрипт демо 26.06.26 (`docs/ascan/meeting-2026-06-26-digest.md`),
карта функционала Аскан по мануалу 2.5 (`docs/ascan/ascan-functional-map.md`), инвентаризация текущего кода.

## 1. Стратегическая рамка

Аскан предложили заказчику вместо нашей CRM. Наша позиция сильнее, чем кажется:

| Возражение против Аскан (с их же демо) | Наш ответ |
|---|---|
| В **KZ-версии Omnicomm-интеграции не будет** (только Wialon); интеграция односторонняя — объекты/блокировки пушатся, **телеметрия не забирается** | Omnicomm-нативная интеграция + забор телеметрии (топливо, пробег, активность) — киллер-фича |
| **Казахстанской сборки не существует** (УТ KZ = 11.4, вендор на 11.5) | Мы KZ-нативны: тенге, ИС ЭСФ через 1С:БУХ KZ, RU/KK UI |
| Автоблокировка должников — есть в РФ-версии, в KZ-сборку не попадёт | Автоблокировка в Omnicomm по правилам |

> **Поправка по аудиту демо (06.07.2026, `download/ascan_audit/`):** в российской демо-базе
> интеграция с Omnicomm Online **есть**, включая режим «блокировать при задолженности автоматически»
> (карточка учётной записи СМ, скрин `ascan_29`). Слова на встрече «обмена с Omnicomm нет» относились
> к KZ-версии и к забору данных с оборудования. Аргумент против Аскан сужается, но остаётся:
> KZ-сборка будет без Omnicomm, и телеметрию Аскан не тянет ни в одной версии.
| Вход ~0.5–0.7 млн ₽ (лицензии 1С + поставка) + 130–210к ₽/год подписки; доработки 4500 ₽/ч, модули закрыты | Web-приложение без лицензий, доработки свои |
| Нет: push технику, сводного отчёта по оборудованию, командировок, ЗП с порогом, автовыдачи оборудования | Всё это — прямые боли заказчика с встречи → делаем |

Что у Аскан реально сильно и что копируем как эталон: **биллинг** (3 типа начислений, посуточный
расчёт, иерархия тарифов, массовые расчётные документы) и **сервисный контур**
(заявка → график нарядов drag-and-drop → акт ТО как центральный документ).

## 2. Что уже есть (прод crm-backend, Express+SQLite, :3026)

Клиенты, объекты, заявки (16 типов, история, фото, бизнес-правила закрытия), выезды монтажников с
гео-этапами, оборудование/склады/SIM (базово), продажи → наряды → акты ТО → абонплата
(планы, инвойсы, cron, авто-блокировка-заглушка), звонки/сообщения, дашборд, RBAC 7 ролей.

Чего нет: реальной интеграции с Omnicomm API, фронтенда (только HTML-прототипы), PostgreSQL,
мобильного РМ техника, тарифной иерархии и посуточного биллинга, массового формирования и рассылки
документов, тестирования/ремонта/БУ, сдельной ЗП, отчётов с Excel-выгрузкой.

## 3. Архитектурное решение

**Единое Next.js 16 приложение (фронт + API routes) + PostgreSQL 16.** Обоснование:
- текущий бэкенд — SQLite без фронта; фронт всё равно писать с нуля, а это ~половина оставшейся работы;
- в `liftplatform-omnicomm/prisma/*.sql` уже спроектированы PG-схемы (тарифы, скидки, payroll,
  движения SIM/оборудования, RMA) — берём как основу миграций, отвязав от LiftPlatform;
- бизнес-логику `crm-backend` (правила статусов, поток продажа→акт→биллинг, smoke-сценарии)
  переносим как спецификацию, не выбрасываем.

Деплой по стандарту: VPS + cron+flock + Cloudflare Tunnel на crm-omnicomm.technokod.kz.
PWA для техника вместо Android-приложения (у Аскан — малотиражный Android).

## 3.1 Адаптация под РК (сквозной слой)

Это главный структурный козырь против Аскан: у них РК-локализации не существует, у нас она — основа.

**Налоги и документы (новый НК РК с 01.01.2026):**
- НДС **16%** (было 12%); ставка — настраиваемый параметр с датой действия, не константа
  (переходные кейсы: ставка определяется датой оборота, не датой договора/оплаты).
- Признак плательщика НДС у организации-продавца (порог регистрации снижен до 10 000 МРП —
  контрагенты-неплательщики тоже встречаются); цены с НДС/без НДС в тарифах.
- Первичка РК: **счёт на оплату**, **АВР (форма Р-1)** — обязателен по работам/услугам,
  **накладная на отпуск запасов (З-2)** — по оборудованию, доверенность. Все — двуязычные (KK/RU).
- **ЭСФ обязателен, ЭАВР добровольный**, но по услугам ЭСФ выписывается на основании АВР →
  целевая цепочка: CRM формирует АВР/накладные → выгрузка в 1С:Бухгалтерию для Казахстана →
  бухгалтер выпускает ЭСФ/ЭАВР в ИС ЭСФ с ЭЦП НУЦ РК (как у заказчика сейчас).
  Прямая интеграция с API ИС ЭСФ — фаза 2, после обкатки (подпись КалканКрипт — отдельный риск).
- **СНТ**: телематическое оборудование — импорт; проверить с бухгалтером заказчика, попадает ли
  номенклатура под сопроводительные накладные при реализации/перемещении.

**Реквизиты и справочники:**
- Контрагенты: БИН/ИИН (есть), **ИИК (IBAN KZ…), БИК банка, Кбе**, юр. форма (ТОО/ИП/ГУ/КГП),
  резидентство. Валюта — тенге; суммы прописью на печатных формах — на двух языках.
- Операторы связи для SIM: Beeline KZ, Kcell, Tele2/Altel, izi — предзаполненный справочник
  с тарифными планами M2M.
- Часовой пояс единый Asia/Almaty (UTC+5) — во всех расчётах «посуточно» считать по нему.

**Бюджетники/муниципалы** (на демо Аскан спрашивали неслучайно — порядок отгрузки «до установки»):
- Сценарий «счёт и закрывающие до монтажа» + реквизиты гос. учреждений (ГУ, КГП), договор по
  госзакупу с номером на портале goszakup.gov.kz в карточке договора.

**Локализация и данные:**
- UI RU/KK (i18n с первого этапа, а не ретрофитом), печатные формы и e-mail-шаблоны двуязычные.
- Персональные данные — хранение на серверах в РК (KZ VPS), это же аргумент против облака
  Аскан на серверах РФ.

## 4. Этапы

### Этап 0 — Фундамент (≈1 неделя)
- Next.js 16 + PG 16, миграции: перенос 20 таблиц crm-backend + доработки из liftplatform-SQL
  (UUID, TIMESTAMPTZ, снейк-кейс).
- Auth (JWT+кука), RBAC 7 ролей, аудит-лог. Каркас i18n (RU/KK) сразу.
- Базовые CRUD-экраны: клиенты (партнёр/контрагент, БИН/ИИН, ИИК/БИК/Кбе, юр. форма, признак
  плательщика НДС), объекты (ТС: марка, госномер), оборудование (серии/IMEI), SIM (справочник
  операторов РК), склады.
- Сквозной паттерн «каждый список → выгрузка Excel» (боль №2 с встречи) — один компонент/хелпер.
- Миграция данных прода SQLite → PG.

### Этап 1 — Интеграция Omnicomm Online (≈1.5 недели) — киллер-фича
- Справочник телематических серверов (URL, токен/логин), health-проба перед операциями
  (копия omnicomm хрупкая — известная гоча).
- CRM — master: активация оборудования → создание объекта в Omnicomm; консервация → «приём данных
  выключен»; отключение → двухэтапное удаление (список «к удалению» + отдельное право).
- Импорт существующей абонбазы: чтение объектов из Omnicomm → сопоставление/создание клиентов,
  объектов, оборудования (парсинг марки/госномера из имени — как у Аскан).
- Регистр истории состояний оборудования (вкл/выкл с датами) — фундамент посуточного биллинга.
- **Автоблокировка должников**: правила (дни отсрочки для аванса/кредита, допустимый долг),
  предупреждение (email + сообщение), блокировка/разблокировка в Omnicomm, журнал.
- Резерв настроек объекта перед удалением (экспорт профиля), восстановление.

### Этап 2 — Биллинг v2 (≈2 недели) — «главный интерес» заказчика
- Три типа начислений: по активному оборудованию (посуточно из истории состояний), подписки, разовые.
- Иерархия тарифов: объект > клиент > категория обслуживания > по умолчанию; «Не начислять»;
  тарифные планы; скидки суммой до исчерпания.
- Схемы расчётов per-клиент: авансовая (счёт в начале месяца, акт в конце: начисления + доначисление
  − скидки − предоплата) и кредитная (всё в конце месяца).
- НДС 16% как параметр с датой действия; тарифы в тенге с НДС/без НДС.
- Массовое формирование расчётных документов с отборами; PDF (двуязычные формы РК): счёт на оплату +
  **расшифровка абонплаты по объектам** + АВР (Р-1); накладная З-2 по оборудованию;
  e-mail-рассылка очередью партиями.
- Ведомость расчётов (долги/авансы) → источник для автоблокировки (этап 1 замыкается).
- Выгрузка реализаций/АВР в 1С:Бухгалтерию для Казахстана (файл/фид — роут уже задуман) →
  бухгалтер выпускает ЭСФ/ЭАВР в ИС ЭСФ как сейчас. Прямой API ИС ЭСФ — в бэклог после обкатки.
- Сценарий бюджетников: счёт и закрывающие «до установки», реквизиты ГУ/КГП, номер договора госзакупа.

**Контрольная точка: этапы 0–2 = демо заказчику.** Интеграция + биллинг закрывают главный интерес
встречи и оба главных возражения против Аскан.

### Этап 3 — Сервисный контур до паритета (≈2 недели)
- **График заказ-нарядов**: drag-and-drop нераспределённых заявок на шкалу время × техники,
  многодневные периоды, несколько исполнителей, автоподстановка адреса по прошлым выездам.
- Акт ТО — центральный документ: действия Установка/Замена/Демонтаж/Диагностика/Услуга; матрица
  оснований → движения оборудования; материалы по нормам расхода (авто-списание); фотофиксация с
  местами из вида работ; установка/снятие SIM (контроль max SIM); результат «требуется доработка»
  → новый наряд; акты из PWA приходят «в подготовке» → офис закрывает (право на самозакрытие).
- Тестирование оборудования: заказ → виртуальный склад → завершение (продажа → заказ клиента /
  отказ → заявка на демонтаж); отчёт с днями.
- Ремонт/замена: приём от клиента (всегда БУ, долг перед клиентом), передача поставщику, получение,
  выдача/подмена; состояния Новое/БУ, подменный фонд.
- **Автоперемещение оборудования на техника** при назначении наряда (боль №6, у Аскан нет).
- **Командировки в наряде**: период, транспорт, расходы → в стоимость/себестоимость (боль №3, у Аскан нет).

### Этап 4 — PWA техника-установщика (≈1.5 недели)
Референс — «Рабочее место техника» из аудита (роль ДемоТехникУстановщик, PDF `_full`, раздел F):
урезанный АРМ-лаунчер поверх тех же документов, акт — не вкладки, а один прокручиваемый лист.
- Наряды на сегодня/неделю (только свои, открытые), маршрут, статусы этапов с гео (visits уже есть).
- Оформление акта в поле одним листом со сворачиваемыми секциями: оборудование —
  **установленное И снятое** (сценарий замены: серии обоих приборов), SIM, материалы, фото с камеры,
  **подпись клиента на экране** + «подписант от клиента»; печать «наряд с подписями».
- Остатки на своём складе (вкл. БУ), остатки SIM по местам размещения, свои ЗП-начисления.
- **Push**: web-push + дубль в Telegram (боль №9 — «вместо распечатанной бумажки»).
- Гоча: версионирование SW-кеша на каждую сборку.

### Этап 5 — Сдельная ЗП (≈1 неделя)
- Категории исполнителей, расценки (исполнитель > категория > умолчание), компенсации, удержания.
- **Схема «оклад за норму N монтажей + сделка сверх порога»** (боль №13, у Аскан только «в теории»).
- Расчёт за месяц/полмесяца, ведомость план/факт, выгрузка в Excel для бухгалтерии.

### Этап 6 — Техподдержка и коммуникации (≈1 неделя)
- Тикеты: решено удалённо / → авто-создание заявок ТО по одной на объект / отклонено;
  авто-закрытие тикета после выполнения всех заявок ТО.
- Каналы: e-mail, Telegram; WhatsApp — через существующий Baileys-мост (anti-ban гейты).
- Журнал звонков (вебхук телефонии — приёмник уже есть).

### Этап 7 — Отчёты и полировка (≈1 неделя)
- **Сводный отчёт по всему оборудованию** во всех размещениях: склад / техники / клиенты / поставщик /
  тестирование, с расшифровкой до документа (боль №10 — прямой запрос руководства).
- Отчёты с днями: товары на тестировании, принятое от клиентов, у поставщика.
- Загруженность исполнителей план/факт; анализ активности оборудования; дашборд руководителя.
- Добивка казахской локализации: перевод всех экранов/шаблонов (каркас i18n с этапа 0),
  вычитка форм; суммы прописью KK/RU на печатных формах.

## 5. Оценка

~9–11 недель календарно (solo + многоагентная разработка). Демо-веха после этапов 0–2 — ~4–5 недель.
Порядок этапов = приоритеты заказчика с встречи: интеграция и биллинг сначала, паритет сервисного
контура потом, «вау»-дифференциаторы (PWA+push, сводный отчёт) — до финального показа.

## 6. Использовать демо Аскан (2 недели, до ~14.07)

База открывается браузером: https://app.ascan.su/utmt-ti-0000-00-demo-omnikomm-alyans-kz

## 7. Риски

1. **Биллинг — самая ёмкая часть** (посуточные пересчёты, доначисления, смены тарифа среди месяца).
   Митигируем: формулы из мануала Аскан + табличные тест-кейсы до кода.
2. **Хрупкость учётки Omnicomm** (projectkap деградирует) — health-проба и ретраи во всех sync-джобах;
   у заказчика будет своя учётка — риск ниже.
3. **Миграция прод-данных** SQLite→PG — прогнать на копии, ресинк секвенций.
4. **Scope creep** — держимся списка болей из встречи; всё, чего в списке нет, — в бэклог после этапа 7.
5. Демо-доступ Аскан истекает ~14.07 — референсы снять в первые дни (продление через Евгения).
6. **НК РК-2026 свежий** (НДС 16%, пороги, переходные обороты) — макеты счёта/АВР/накладной и
   логику НДС провалидировать с бухгалтером заказчика до конца этапа 2; туда же — вопрос СНТ
   по импортному оборудованию.
